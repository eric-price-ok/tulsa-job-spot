# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Tulsa Job Spot is a forkable, self-hosted job board for local communities. It is designed for single-server deployment with minimal dependencies and no dark patterns. Stack: **FastAPI + SQLAlchemy 2.0 (async) + PostgreSQL 16 + Jinja2 + HTMX + ARQ (Redis-backed workers)**.

## Commands

All dev operations run inside Docker containers. The app uses uvicorn with `--reload` in development.

```bash
# Start the full stack (app, worker, db, redis, caddy)
docker compose up -d

# View logs
docker compose logs -f app
docker compose logs -f worker

# Apply database migrations
docker compose exec app alembic upgrade head

# Create a new migration (after editing models)
docker compose exec app alembic revision --autogenerate -m "description"

# Seed reference data
docker compose exec app python -m app.scripts.seed
```

### Testing

The test suite requires a separate test database. Create it once:

```bash
docker compose exec db psql -U tulsajobspot postgres -c "DROP DATABASE IF EXISTS tulsajobspot_test;"
docker compose exec db psql -U tulsajobspot postgres -c "CREATE DATABASE tulsajobspot_test;"
docker compose exec db bash -c "pg_dump -U tulsajobspot tulsajobspot | psql -U tulsajobspot tulsajobspot_test"
```

Run tests:

```bash
# All tests
docker compose exec -e DATABASE_URL=postgresql+asyncpg://tulsajobspot:tulsajobspot@db:5432/tulsajobspot_test app pytest -v

# Single test file
docker compose exec -e DATABASE_URL=postgresql+asyncpg://tulsajobspot:tulsajobspot@db:5432/tulsajobspot_test app pytest tests/test_security.py -v

# Single test by name
docker compose exec -e DATABASE_URL=postgresql+asyncpg://tulsajobspot:tulsajobspot@db:5432/tulsajobspot_test app pytest tests/test_security.py::test_slug_basic -v
```

See `docs/running-tests.md` for the full testing guide.

## Architecture

### Request Lifecycle

1. **Caddy** (reverse proxy, auto HTTPS) → forwards to `app:8000`
2. **FastAPI middlewares** in `app/main.py`: session cookies, proxy headers, security headers
3. **Router** in `app/routers/` handles the request
4. **Dependencies** in `app/dependencies.py` inject DB session and current user
5. **SQLAlchemy async session** fetches/writes data
6. **Jinja2 template** renders server-side HTML (or JSON for API endpoints)
7. HTMX partials are returned for dynamic interactions without a full page reload

### Key Directories

- `app/models/` — SQLAlchemy ORM models (source of truth; migrations generated from these)
- `app/routers/` — FastAPI route handlers grouped by domain (`auth`, `jobs`, `companies`, `admin/`, `moderator/`)
- `app/templates/` — Jinja2 HTML; partials used for HTMX swaps live alongside full pages
- `app/workers/` — ARQ background job functions and `WorkerSettings` cron config
- `app/scrapers/` — Scraper implementations; each uses Claude API for field extraction
- `migrations/` — Alembic migration files; **always apply after pulling schema changes**
- `docs/` — Architecture, deployment runbook, and deferred items

### Auth & Roles

OAuth-only (no passwords). Five providers configurable via `.env` (Google, LinkedIn, GitHub, Microsoft, Facebook). Session state is stored in signed httponly cookies via `SessionMiddleware`.

Two role axes:
- **Site-level**: boolean flags `is_admin` / `is_moderator` on the `User` model
- **Company-scoped**: `user_company_roles` junction table with `company_admin` / `job_poster` roles

Dependencies that enforce roles: `require_user()`, `require_moderator()`, `require_admin()` in `app/dependencies.py`.

### Job Listing Workflow

- **Employer-posted**: created with `approved=False`, enter moderator queue, approval fires notification email via ARQ
- **Scraped**: ARQ worker fetches from external board → Claude API extracts structured fields → deduplicated via `scraping_hash` → auto-approved

### Background Jobs (ARQ)

The `worker` service runs `arq app.workers.worker.WorkerSettings`. Job functions are defined in `app/workers/` and registered in `WorkerSettings`. Enqueue with `await queue.enqueue_job("function_name", arg1, arg2)`.

### Slug Generation (`app/utils.py`)

Company slugs are generated by `generate_slug()`, which strips common legal suffixes (LLC, Inc, Corp, Ltd, etc.) before slugifying. Collision handling appends a numeric suffix. Do not change the stripping rules without reading `docs/` and the feedback on slug design decisions.

### Configuration (`app/config.py`)

All settings loaded from `.env` via pydantic-settings into a `settings` singleton. Copy `.env.example` to `.env` to get started. On production, `.env` must contain `COMPOSE_FILE=docker-compose.yml:docker-compose.prod.yml` to layer the prod overrides.

## Data Model Notes

- **Soft deletes everywhere**: records are archived via status flags (`archived`, `approved`, etc.), never hard-deleted
- `scraping_hash` on job listings prevents duplicate imports from scrapers
- `oauth_provider` + `oauth_subject` is the unique identity for users (no passwords stored)
- All timestamps are UTC; `created_at` / `updated_at` columns are on most tables

## Deployment

Production runs Gunicorn (4 workers) instead of Uvicorn. See `docs/deployment-runbook.md` for VPS setup. For a fresh deployment, `setup.sh` bootstraps Docker, `.env`, initial migration, and seed data in one pass.
